# 从零搭建现代 Android 模块化架构实战

在日益复杂的移动端开发环境中，一个灵活、可扩展且高内聚低耦合的架构，是支撑应用持续演进的基石。最近我花时间重构了手头上的 Android 项目，结合现代 Android 开发规范，搭建了一套轻量级模块化架构。

今天，我将从**全局分层**、**模块化解耦**、**UI 及异步流管控**，以及**工程化约束**四个维度，和大家分享这套架构的落地实践。

---

## 纵览全局：清晰的四层架构体系

在一个中大型应用中，如果没有明确的边界，代码很容易演变成“意大利面条”。为了解决这个问题，我将项目自底向上划分为四层：

1.  **地基模块 (Core 层)**：最底层的基石，完全无关具体业务。它包含公共常量库 (`core_base`)、网络请求引擎 (`core_network`)、通用 UI 控件与基调 (`core_ui`)、核心工具类 (`core_util`)、全局路由体系 (`core_router`)，以及**本地二进制集中转发所用的 `core_aar`** 等。无论上面跑什么业务，Core 层都岿然不动。
2.  **通用组件 (Component 层)**：提供跨越特定业务线的通用能力。比如，直播功能 (`component_live`)、美颜模块 (`component_beauty`) 或是支付能力 (`component_pay`)。它们比 Core 层更贴近业务，但又由于其泛用性，独立于单一的 Feature 之外。
3.  **特性业务 (Feature 层)**：按具体的业务线垂直切分。这里的代码具有高度业务属性，例如项目的核心体验都在这里承载：首页动态 (`feature_home`)、用户资料 (`feature_user`) 以及登录控制 (`feature_login`)。通过切分，可以实现并行开发与独立测试。
4.  **入口外壳 (App 层)**：所有子模块最终汇聚的枢纽，它不含具体业务逻辑代码，只负责引入所需模块，进行全局配置（如初始化 SDK）并打出最终可用的 APK。

通过依赖倒置与单向依赖规则，下层模块（Core、Component）决不允许反向依赖上层模块（Feature）。这意味着底层功能可以被无限复用。

## “大道至简”：为什么我们选择自研极简路由？

关于模块化，绕不开的话题就是**页面通信**。

市面上有很多成熟庞大的路由框架（比如 ARouter）。但在实际开发中，我们发现这些重型框架往往伴随着代价：
*   **编译期耗时增加**：重度依赖 KAPT / KSP 注解处理器，在多模块下每次增量编译都要重新扫描生成大量类文件。
*   **黑盒侵入性强**：出了 Bug 难以追踪调用栈，甚至为了兼容不断升级的 AGP 版本（如我们现在使用的 AGP 8.13+）经常出现莫名其妙的兼容报错。
*   **大炮打蚊子**：绝大部分项目80%以上的需求，仅仅只是简单的“跨模块打开一个 Activity 或者获取一个 Fragment”。

既然需求本质如此简单，为什么我们要忍受重型框架带来的心智负担？基于“如无必要，勿增实体”的开发哲学，我在 `core_router` 中实现了一个**不到一百行代码**的极简路由与服务发现方案。

### 1. Router：纯净的内存映射映射
抛弃了所有编译时注解，一切回到最可控的运行时内存注册。各个业务方自己在组件初始化阶段注入：

```kotlin
// 注册 Activity 路线
Router.add(Route.activity("/login/main", LoginActivity::class))
// 注册 Fragment 路线
Router.add(Route.fragment("/home/fragment") { HomeFragment() })
```

**跳转端同样干净利落。** 我们充分利用了 Kotlin 的高阶函数，通过 DSL 的方式优雅地透传 `Intent`，不额外造传参的数据结构轮子：
```kotlin
Router.open("/login/main", context) {
    putExtra("key", "value") // 还是你最熟悉的 Intent 用法
}
```

### 2. ServiceManager：返璞归真的面向接口编程
不仅要解耦页面，业务逻辑也要解耦。我们没有搞高大上的依赖注入容器来实现跨层调用，而是回归了最原始也是最轻巧的方式：

一处声明标准（底层的接口定义）：
```kotlin
interface IUserService { fun getUserName(): String }
```
一处注册实现（具体业务模块的实现接入）：
```kotlin
ServiceManager.register(IUserService::class.java, UserServiceImpl())
```
一处按需提取（任意需要查询用户的模块）：
```kotlin
val userService = ServiceManager.get(IUserService::class.java)
```

**简陋吗？是的，代码极其短小。但它脆弱吗？绝不。**
这种方案将黑盒彻底变成了白盒，配合 Kotlin 的高阶函数，不仅满足了我们跨模块跳转、接口服务注入的所有真实需求，更是彻底把增量编译的性能压榨到了极限——因为压根不需要 KAPT 处理任何路由注解。

## 现代开发利器：响应式 UI 与并发流

架构搭建好了，在每个模块内部的代码怎么写才漂亮？我们同样引入了一系列现代 Android 的“最佳实践”：

1.  **MVVM / MVI 模式**：告别 Activity 包揽一切的旧时代。由 **ViewModel** 来统管界面状态，使其能够抵御屏幕旋转等配置变更导致的数据重建摧残。配合 **ViewBinding**，彻底抛弃臃肿易错的 `findViewById`。
2.  **Kotlin 协程与 Flow**：抛弃了杂乱无章的回调地狱（Callback Hell）。项目中所有异步操作、网络请求全部交给 Kotlin Coroutines 与 Flow 处理，代码结构更像同步代码般清晰简单，大大降低了引发内存泄漏的风险。
3.  **依赖注入控制**：引入现代化的依赖注入机制（如 Hilt / Koin），解决对象创建树状依赖过密的问题。这让我们的代码更方便替换实现，单元测试写起来也如丝般顺滑。

## 铁腕统治：工程化约束保证团队不跑偏

优秀的架构不能只停留在纸面，更要落地到代码里的每一个犄角旮旯。项目采用严格的规则体系：

*   **Version Catalog 依赖统管**：通过 `gradle/libs.versions.toml`，项目的版本库管理不再散发在几十个 `build.gradle` 文件里。改一处，全盘统一，彻底杀死了“为何你的依赖行，拉到我这里就冲突”的玄学 Bug。
*   **严密的代码审查**：集成 `Lint` 与 `Detekt`，所有的 PR 必须通过强制检查。布局命名规范化（`activity_`，`item_` 前缀）、硬编码警告，一切不规范都被挡在主分支门外。
*   **极致编译体验**：在 `gradle.properties` 开启了 Configuration Cache 与 Build Cache，结合我们的多模块结构，让曾经动辄数十秒的增量编译，下降到了秒级极速响应。

---

好的架构不是一蹴而就的，而是要在“克制”与“满足业务”之间寻找平衡点。这套“轻量级纯粹代码路由 + 现代工程化规范”的模块化实践，舍弃了沉重的黑盒框架，换回了秒级的极速编译体验和清晰透明的代码调用栈。

如果你正面临庞大路由框架带来的龟速编译之痛，或者是团队多线并行开发的极度耦合，不妨考虑“大道至简”，也亲手写一个不超过百行的架构引擎兜底吧！
